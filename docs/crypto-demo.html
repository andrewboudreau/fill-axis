Signed Game Chain (ECDSA) — Tamper-Evident Audit Log

Beyond the commitment, every game action is signed with the player's private key and chained via prevHash. After the game, anyone with both public keys can prove no moves were inserted, deleted, or modified. Private keys never leave the browser.

Async Fair Exchange (for turn-based / offline play)

In a real-time game both players are online at reveal — disconnect = forfeit solves the problem. For async play (email chess style) you need a different protocol. This is the "both deposit" idea you described.

P2->>S: poll — is sealed₁ there? Note over P1,P2: Both see both blobs exist. Safe to exchange decrypt keys. P1-->>P2: send P1 private key (or decrypt key) P2-->>P1: send P2 private key (or decrypt key) P1->>P1: decrypt sealed₂ with P2 key → verify commitment P2->>P2: decrypt sealed₁ with P1 key → verify commitment

Why this works

If P1 uploads sealed₁ and P2 never uploads — P1 simply doesn't share their decrypt key. Neither player learns anything.

Once both sealed blobs exist, each player knows the other is committed. Sharing the decrypt key reveals nothing beyond what's already locked.

The blob store is untrusted — it sees only encrypted ciphertext. It cannot read either board.

Current status: Real-time P2P (the game as built) doesn't need this — PeerJS + forfeit-on-disconnect is sufficient. This protocol is documented for a future async mode.

How-To: Implementing in a P2P Game

1
Generate a salt at game start — before any pieces are placed. generateSalt() returns 32 random bytes as hex. Keep it in memory only — never log it, never send it yet.
2
Place pieces normally. The board is hidden from your opponent (sent as TILE_HIDDEN over PeerJS). Nothing crypto-related happens during placement.
3
When all pieces are placed: commit. Compute commitment = SHA256(boardStateString(pieces) + salt). Send {type:'COMMIT', hash: commitment} over PeerJS. Wait for the opponent's commitment before proceeding.
4
Once both commitments are received: reveal. Send {type:'REVEAL', pieces: myPieces, salt: mySalt}. Only now is the salt exposed.
5
Verify the opponent's reveal. Compute SHA256(boardStateString(theirPieces) + theirSalt) and compare to the commitment hash they sent in step 3. If it matches: fair game, run battle. If not: ❌ CHEAT DETECTED, game is void.
6
The critical ordering rule: you must have the opponent's commitment before you reveal your salt. Once they have your salt they can recompute your hash — but that's fine, because they already gave you their commitment, so they can't change their board either.
